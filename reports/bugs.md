## 🔴 عالية الخطورة

### 1. 🐛 نشر الكورس لا يعمل (`is_published` غير فعّال)

| البند | القيمة |
|---|---|
| **المكان** | `PUT /api/courses/:id` + زر "نشر للطلاب" في الـ UI |
| **الأعراض** | إرسال `is_published: true` عبر API يُعيد `is_published: false` | | |
| **السيناريو** | `PUT /api/courses/159` ← `{is_published: true}` ← الـ response يعيد `false` |
| **التأثير** | الكورس لا يظهر للطلاب أبداً مهما ضغطت "نشر للطلاب" |
| **المتوقع** | `is_published` يجب أن يتغير فعلياً في الـ DB ويظهر للطلاب |

**ملاحظة:** في وقت سابق من الاختبار كان فيه 5 كورسات منشورة و2 غير منشورة (من ضمنها الكورس اللي أنا أضفته)،
وبعد ريلود الصفحة اختفى الكورس اللي أضفته وخلال اختبار لاحق كان فيه 5 منشور و1 مسودة فقط.

### 2. 🐛 API تسجيل دخول الطالب معقّد (يحتاج Slug + Device ID)

| البند | القيمة |
|---|---|
| **المكان** | `POST /api/auth/login` (Login الطالب) |
| **الأعراض** | Login الطالب عبر API يفشل بدون `X-Tenant-Slug` header و `device_id` في الـ body |
| **السيناريو 1** | POST عادي ببيانات std_ali ← `401 "بيانات الدخول غير صحيحة"` رغم صحة البيانات |
| **السيناريو 2** | POST مع `X-Tenant-Slug` ← `400 "DEVICE_ID_REQUIRED"` |
| **السيناريو 3** | POST مع slug + device_id جديد ← `403 "NEW_DEVICE_BLOCKED"` |
| **التأثير** | أي اختبار API مباشر للطالب سيفشل برسائل غامضة أو متطلبات مخفية |
| **المتوقع** | يجب أن يعمل login أساسي من غير device_id على الأقل في بيئة التطوير، أو رسالة خطأ توضح المتطلبات بدقة |

**ملاحظة:**
- تسجيل الدخول عبر المتصفح (UI) شغال تماماً — دخلت بـ std_ali/123456 من غير مشاكل
- المشكلة بس في API المباشر — الـ device tracking security قوي جداً لدرجة إنه بيمنع API testing

## 🟡 متوسطة الخطورة

### 4. 🐛 الـ Modals غير مرئية في Accessibility Tree

| البند | القيمة |
|---|---|
| **المكان** | مودالات: إضافة طالب، إضافة كورس، إضافة اختبار |
| **الأعراض** | المودالات معمولة بـ React Portal ومش باينة في الـ accessibility tree أبداً |
| **السيناريو** | فتح مودال إضافة طالب → الـ snapshot يظهر الصفحة الخلفية فقط رغم وجود المودال في الـ DOM |
| **التأثير** | - صعوبة في الاختبار الآلي
- مشاكل في الـ screen readers (إعاقات بصرية)
- أدوات الـ E2E testing (Cypress/Playwright) ممكن تواجه مشاكل |
| **المتوقع** | المودال يظهر في accessibility tree أو على الأقل يكون في `role="dialog"` مع `aria-modal="true"` |

**التحقق:**
```javascript
// المودال موجود في DOM (2 modals, 1 overlay)
document.querySelectorAll('[class*="modal" i]').length  // ← 2
document.querySelectorAll('[class*="overlay" i]').length // ← 1
// لكن مش باين في accessibility snapshot
```

### 7. 🐛 بعض الاختبارات بتظهر "✓ نجاح" للطالب قبل أدائها

| البند | القيمة |
|---|---|
| **المكان** | صفحة اختبارات الطالب |

`✓ نجاح` في الـ accessibility tree رغم إن الطالب لسه مأديهاش |
| **التحقق** | الـ DOM الحقيقي بيظهر `⏳ قريباً` للاختبارات المستقبلية — المشكلة في الـ accessibility rendering مش الـ logic |
| **السيناريو** | دخول الطالب لصفحة اختبارات ← شايف ✓ نجاح على اختبارات لسه مأديهاش |
| **التأثير** | confusion للطالب — ممكن يفتكر إنه خلص اختبارات وهو لسه ما عملهاش |
| **المتوقع** | status يظهر `⏳ قريباً` أو `غير مبدوء` للاختبارات اللي الطالب مأديهاش، في كل حتة في الـ DOM |
